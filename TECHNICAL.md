# VKEncrypt — техническая документация

> Этот документ — справочник для разработчиков и продвинутых пользователей. Если ты просто хочешь шифровать переписку в ВК — тебе в [README.md](README.md).

---

## Что это

VKEncrypt — end-to-end шифрование переписки ВКонтакте через AES-256-GCM. Состоит из двух частей:

- **`extension/`** — userscript для браузера (Tampermonkey / Userscripts / Kiwi).
- **`bot/`** — VK-бот на Python, работает как echo-шифратор для проверки ключей.

Ключи никогда не передаются ВКонтакте — только шифр вида `ENC[k1:base64(iv ‖ ciphertext ‖ tag)]`.

Расширение и бот используют **одну и ту же модель ключей**: seed-фраза через PBKDF2 → k1..k4, плюс пользовательские 64-hex ключи. Ввести фразу / ключ можно либо в UI userscript'а, либо командами в чате с ботом.

## Подробно про расширение

### Что умеет

- **🔒 Шифрует** исходящие сообщения — кнопка в поле ввода или автоматически при отправке (Enter, клик Send). Shift+Enter = перенос строки.
- **🔓 Расшифровывает** входящие прямо в чате, с переключателем `[шифр]/[текст]`. Тумблер расшифровки в меню.
- **🔑 Управляет ключами** — seed-фраза → k1..k4, пользовательские 64-hex, временный `@temp`.
- **⚡ Генерирует новые 256-битные ключи** в один клик с автокопированием.
- **😀 Emoji-шифротекст** — опциональная замена Base64 на 64-emoji алфавит (маркер `emj.`, паддинг 🟰).
- Работает в **любом чате ВК**: личном, групповом, беседе, на `vk.com` и `vk.ru`, в старой и новой версии интерфейса.

### Формат шифра

```text
ENC[k1:base64( iv ‖ ciphertext ‖ tag )]
```

| Поле | Размер | Что это |
|------|--------|---------|
| `k1` | 2-8 байт | ID слота ключа (`k1`..`k4`, кастомный id, или `@temp`) |
| `iv` | 12 байт | Случайный nonce, новый для каждого сообщения |
| `ciphertext` | переменный | Зашифрованный AES-256-GCM текст |
| `tag` | 16 байт | GCM-тег целостности (проверяется автоматически) |

Если включён emoji-режим, после двоеточия идёт `emj.<emoji-алфавит>` вместо Base64. На дешифровке префикс `emj.` снимается автоматически.

Если сообщение повреждено или подменено — получатель увидит `[❌ Ошибка расшифровки: ...]`.

### Криптография

| | |
|---|---|
| Алгоритм | **AES-256-GCM** (Web Crypto API, нативно в браузере) |
| Длина ключа | 256 бит (32 байта, 64 hex-символа) |
| IV/Nonce | 12 байт, новый для каждого сообщения |
| Tag | 16 байт, проверяется при расшифровке |
| KDF | PBKDF2-SHA256, 250 000 итераций, salt `vk-p2p-aes-gcm-v1`, dklen 1024 бит → 4 × 32 байта |
| Внешние библиотеки | **Нет** — только `crypto.subtle` |

KDF-параметры должны **байт-в-байт совпадать** с теми, что использует бот (`bot/main.py::KDF_SALT`, `KDF_ITERATIONS`, `KDF_DKLEN`). При совпадении одна и та же фраза даёт одинаковые k1..k4 на обеих сторонах.

Работает полностью офлайн, ничего никуда не отправляется.

### Кнопки в поле ввода

| Кнопка | Что делает |
|--------|------------|
| 🔒 | Зашифровать набранный текст. Прячется при включённом автошифровании. |
| 🔑 | Открыть меню ключей и настроек. |

Если ключей нет — клик по любой из них открывает seed-модалку.

### Меню 🔑

| Пункт | Назначение |
|-------|------------|
| `k1..k4` | Производные ключи из seed-фразы |
| `<кастомный id>` | Пользовательские 64-hex ключи, добавленные через «➕ Добавить пользовательский ключ» |
| `@temp` | Временный ключ (только в памяти, исчезает при F5) |
| 🟢 Автошифрование: вкл/выкл | Тумблер автоматического шифрования при отправке |
| 😀/🔤 Emoji-шифротекст: вкл/выкл | Тумблер кодирования Base64 → emoji |
| 👁️/🙈 Расшифровка входящих: вкл/выкл | Тумблер расшифровки в чате |
| ➕ Добавить пользовательский ключ | Модалка с именем и 64 hex |
| ⚡ Сгенерировать временный ключ | Создаёт 256-бит, кладёт в `@temp`, копирует в буфер |
| 🔄 Сменить seed-фразу k1..k4 | Модалка с глазом 👁️ |
| 🧹 Удалить временный ключ | Очистить `@temp` |
| 🗑️ Удалить ключ `<id>` | Для каждого пользовательского ключа |
| 🧨 Сбросить все сохранённые ключи | Очистить DERIVED + CUSTOM, открыть seed-модалку |

ESC закрывает открытое меню.

### Хранение ключей

| Тип | Где живёт | Когда исчезает |
|-----|-----------|----------------|
| **DERIVED** (k1..k4) | `GM_setValue('vk_p2p_derived_keys_v1', ...)` | Когда пользователь сбросит / сменит seed / выключит сохранение |
| **CUSTOM** | `GM_setValue('vk_p2p_custom_keys_v1', ...)` | Когда пользователь удалит ключ / сбросит всё |
| **@temp** | только в памяти страницы | При F5, закрытии вкладки |
| **Settings** (autoEncrypt, emojiCipher, decryptIncoming) | `GM_setValue('vk_p2p_settings_v1', ...)` | Ручной сброс Tampermonkey для скрипта |

**Seed-фраза никогда не сохраняется** — ни в `GM_setValue`, ни в `localStorage`. После первого дерива она отбрасывается.

### Обмен ключами по P2P

**Способ 1: seed-фраза.** Оба собеседника вводят одну и ту же фразу — у обоих будут одинаковые k1..k4. Самый простой путь.

**Способ 2: hex-ключ.** Собеседник А нажимает 🔑 → ⚡ Сгенерировать временный ключ — ключ появляется в `@temp` и **копируется в буфер обмена**. А передаёт ключ Б любым каналом вне ВК (Signal, QR-код, голосом, в Telegram, на бумажке). Б добавляет его через ➕ Добавить пользовательский ключ. Оба выбирают ключ и шифруют сообщения.

### Про дефолтные ключи

В v4+ в коде userscript **нет** `STATIC_KEYS`. Все ключи вводятся пользователем через UI (seed-фраза или hex). Поэтому:

- Свежий install = нет ключей = кнопка 🔑 открывает seed-модалку.
- Нет риска «демо-ключ ушёл в прод» — его физически нет.
- build.sh больше не подставляет ничего из `bot/.env` (там больше нечего подставлять).

### Автообновление

В шапке userscript указаны `@updateURL` и `@downloadURL`, указывающие на `extension/vkencrypt.user.js` в master-ветке репозитория. При каждом запуске Tampermonkey/Userscripts проверяет эту ссылку: если `@version` вырос — пользователю предлагается обновиться в один клик.

`.user.js` — обязательное расширение для Tampermonkey (браузер распознаёт userscript только по нему). `vkencrypt.template.js` — исходник; `build.sh` просто копирует template в `.user.js` и в `dist/`. Подробнее — в [extension/README.md](extension/README.md).

### Версионирование

Бампай `@version` в шапке `vkencrypt.template.js` при любом изменении. Семантическое версионирование:

- **patch** (4.0.x → 4.0.x+1) — мелкие правки, багфиксы.
- **minor** (4.0 → 4.1) — новая фича, обратная совместимость шифра сохраняется.
- **major** (4.x → 5.0) — ломающие изменения формата `ENC[...]` или KDF (не делай без крайней нужды, старые шифры перестанут читаться).

Текущая версия — **4.0**. Переход с 3.x — breaking: модель ключей переделана, никаких `STATIC_KEYS` в коде больше нет.

## Как менять ключи

### Способ 1: сменить seed-фразу (k1..k4)

В userscript'е: 🔑 → 🔄 Сменить seed-фразу k1..k4. Старые сохранённые k1..k4 перезаписываются новыми, пользовательские ключи не трогаются.

В боте: `/setseed <новая фраза>`.

### Способ 2: добавить пользовательский ключ

В userscript'е: 🔑 → ➕ Добавить пользовательский ключ → имя + 64 hex.

В боте: `/setkey <id> <64-hex>`.

### Способ 3: сгенерировать временный ключ (P2P, без правки файлов)

В userscript'е: 🔑 → ⚡ Сгенерировать временный ключ (ключ попадёт в `@temp` и в буфер обмена).

В боте генерации временного ключа нет — бот stateless и не предлагает эфемерные сессии. Используй `/setkey @<id> <64-hex>` если хочется одновременно и в боте.

### Способ 4: синхронизировать ключи между устройствами

1. На устройстве 1: перейди по [установочной ссылке](https://raw.githubusercontent.com/megamen32/vkencrypt/master/extension/vkencrypt.user.js) — установи/обнови userscript.
2. На устройстве 2: то же самое.
3. На обоих устройствах введи одну и ту же seed-фразу.

### Способ 5: hex-ключи через env (для self-hosted бота)

В `bot/.env` можно положить `PRE_SHARED_KEY_K1=...`. Бот подхватит, но **userscript про этот ключ не узнает автоматически** (в v4+ env-инжекции в userscript нет). Чтобы расширение и бот разделили ключ — добавь его в userscript через ➕ с тем же hex и тем же именем слота.

## ВК-бот (опционально)

VKEncrypt-бот — это **echo-сервер с AES-256-GCM**, который проверяет, что твои ключи работают: получает твоё сообщение, расшифровывает (или шифрует) и отправляет обратно. Полезен для отладки, проверки новых ключей и обмена зашифрованными сообщениями «с самим собой».

Бот **не хранит** сообщения и **не видит** открытый текст — работает только с шифром.

### Зачем он нужен

- **Проверить ключи** — отправил `hello`, получил `ENC[k1:...]`. Значит ключ работает.
- **Расшифровать вручную** — если на телефоне нет расширения, можно шифровать через бота.
- **Получить эхо** — отправил шифр боту, он расшифровал и прислал обратно зашифрованным.
- **Делить фразу с расширением** — ввёл `SEED_PHRASE=...` в `.env` (или `/setseed` в чате) → k1..k4 совпадают с userscript'ом.

### Установка

```bash
cd bot
sudo ./deploy.sh
```

Скрипт автоматически:

- создаст `.venv`,
- установит зависимости (`uv sync` или `pip install`),
- сгенерирует `.env` с двумя случайными ключами K1, K2 (если файла ещё нет),
- зарендерит `vkencrypt.service` и положит в `/etc/systemd/system/`,
- запустит бота,
- **соберёт актуальный userscript** в `../extension/dist/`.

После `deploy.sh` открой `.env` и впиши свои `VK_TOKEN` и `MY_USER_ID`. Можно также заменить hex-ключи на `SEED_PHRASE=...` (тогда k1..k4 будут производными).

### Где взять VK_TOKEN

1. Создай сообщество ВК (или используй существующее).
2. Управление → Дополнительно → Работа с API → Ключи доступа → Создать ключ.
3. В настройках сообщества включи **Long Poll API** (Типы событий → Сообщения).

### Где взять MY_USER_ID

[id.vk.com/account](https://id.vk.com/account/#/personal) — там указан ваш числовой ID.

### Управление

```bash
sudo systemctl status vkencrypt         # статус
sudo systemctl restart vkencrypt        # перезапуск (после правки .env)
sudo journalctl -u vkencrypt -f         # логи в реальном времени
```

### Команды в чате с ботом

| Команда | Что делает |
|---------|------------|
| `/setseed <фраза>` | Деривит k1..k4 из фразы, сохраняет в `bot/keys.json` |
| `/delseed` | Удаляет k1..k4 (но не custom-ключи) |
| `/setkey <id> <64-hex>` | Добавляет пользовательский ключ |
| `/delkey <id>` | Удаляет пользовательский ключ |
| `/keys` | Список активных ключей |
| `/help` | Справка |
| `любой текст` | Зашифрует и пришлёт обратно (если ключи есть) |
| `ENC[id:base64...]` | Расшифрует, зашифрует заново, пришлёт обратно |

Команды понимает только `MY_USER_ID`. Ключи из `/setseed` и `/setkey` сохраняются в `bot/keys.json` (chmod 600, gitignored).

### Криптография

Та же, что в расширении: **AES-256-GCM**, **PBKDF2-SHA256 / 250 000 итераций / salt `vk-p2p-aes-gcm-v1`**. Подробности — в [bot/README.md](bot/README.md).

## Безопасность

### Что шифруется

- **Только текст** в поле ввода и входящие текстовые сообщения формата `ENC[...]`.
- GCM-тег автоматически ловит любую подмену — повреждённое сообщение не расшифруется.

### Что НЕ шифруется

- **Метаданные**: кому и когда отправлено — ВКонтакте это знает.
- **Стикеры, фото, видео, голосовые, файлы, геолокация** — шифруется только текстовое поле.
- **Push-уведомления**: превью сообщения может показать начало шифра, если в настройках ВК включён показ текста в уведомлениях.
- **Содержимое ленты, стены, комментариев** — скрипт работает только в чатах.

### Про хранение ключей

- **DERIVED (k1..k4)** — в `GM_setValue` userscript'а (локально в Tampermonkey). Никуда не отправляются.
- **CUSTOM** — там же, в `GM_setValue`.
- **@temp** — только в памяти открытой вкладки `vk.com` / `vk.ru`. Исчезает при F5, не пишется в localStorage/cookies.
- **В `.env` бота** — на сервере, с правами `chmod 600`. Не попадает в git.
- **В `bot/keys.json`** — там же, chmod 600, gitignored.
- **Seed-фраза** — нигде не сохраняется вообще; только в голове пользователя.

### Если компрометирован

Один ключ скомпрометирован — смени seed (создастся новый набор k1..k4) и/или удали скомпрометированный custom-ключ. Старые шифры с этого ключа останутся у того, кто их перехватил.
